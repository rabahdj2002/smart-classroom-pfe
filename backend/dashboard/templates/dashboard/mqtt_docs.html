1) Send Data To Platform (Device → Dashboard)

Publish JSON payloads on the events topic. These payloads update classroom state and session attendance.

Minimum fields

  • classroom_name or classroom_id
  • occupied (true/false)

Common optional fields

  • teacher_rfid or staff_rfid
  • student_rfids array or student_rfid
  • timestamp (ISO datetime)
  • lights_on, door, projector_on
  • smoke_detected, danger_indicator
  • new_session (force session split)

Canonical event example

{
  "classroom_name": "d3",
  "timestamp": "2026-04-17T15:10:05+01:00",
  "teacher_rfid": "123456789123",
  "student_rfids": ["1234567891254", "STU-002"],
  "occupied": true,
  "lights_on": true,
  "door": false,
  "projector_on": true,
  "smoke_detected": false,
  "danger_indicator": false,
  "new_session": false
}
2) Receive Commands From Platform (Dashboard → Device)

Subscribe your microcontroller/service to:

smartclass/classrooms/<classroom_name>/commands

Command payload format

{
  "command": "lights" | "projector" | "smoke_reset",
  "classroom_id": 3,
  "classroom_name": "d3",
  "value": true | false
}

How devices should handle commands

  • lights: set lights relay to value.
  • projector: set projector relay to value.
  • smoke_reset: clear smoke alarm state after local safety checks.
3) Tested CLI Examples

Publish classroom event (device simulation)

mosquitto_pub -h {{ mqtt.broker_host }} -p {{ mqtt.broker_port }} -t smartclass/classrooms/d3/events -m "{\"classroom_name\":\"d3\",\"teacher_rfid\":\"123456789123\",\"occupied\":true,\"lights_on\":true}"

Listen for commands for classroom d3

mosquitto_sub -h {{ mqtt.broker_host }} -p {{ mqtt.broker_port }} -t smartclass/classrooms/d3/commands -v

Publish command manually (for verification)

mosquitto_pub -h {{ mqtt.broker_host }} -p {{ mqtt.broker_port }} -t smartclass/classrooms/d3/commands -m "{\"command\":\"lights\",\"classroom_name\":\"d3\",\"value\":true}"

4) Session And Attendance Behavior
  • If no open session exists for a room, the first valid event opens one automatically.
  • If new_session is true, the current session closes and a new one starts.
  • If teacher RFID changes in the same room, the previous session is closed and replaced.
  • Student RFIDs in payload are attached to the active session for attendance reporting.
  • Auto-finish closes overdue sessions and generates reports based on settings.
5) Security And Authentication

Recommended authentication approach

  • Use dedicated MQTT credentials per environment (test and production).
  • Never store broker passwords in firmware source or public repositories.
  • Rotate credentials regularly and after any suspected device compromise.

Authorization and ACL rules

  • Allow each device to publish only to its own events topic.
  • Allow each device to subscribe only to its own commands topic.
  • Deny wildcard publish permissions for edge devices in production.

Transport security

  • Prefer MQTT over TLS (commonly port 8883) outside trusted local networks.
  • Validate broker certificates on clients when TLS is enabled.
  • Disable anonymous broker access in production environments.
6) Critical Reliability Details

Message handling

  • Use QoS 1 for important state events when delivery confirmation is required.
  • Avoid retained event payloads unless state replay is explicitly intended.
  • Include ISO timestamp values to keep session timing deterministic.

Payload consistency

  • Send explicit booleans for status flags: occupied, lights_on, door, projector_on.
  • Keep classroom names and RFID formats consistent across all devices.
  • Use new_session: true when you intentionally need a session split.

Operational controls

  • Monitor broker availability, reconnect rates, and authentication failures.
  • Configure broker persistence and backups to reduce data loss risk.
  • Keep a mapping of device IDs to classrooms for faster incident response.
7) Troubleshooting
  • No dashboard updates: verify topic path and broker credentials first.
  • Unknown teacher/student: confirm RFID values match database records exactly.
  • Commands not received by device: confirm device is subscribed to the classroom commands topic.
  • Wrong occupancy status: always include explicit occupied in event payload.
  • Time mismatch in sessions/reports: include ISO timestamp in each event.
  • TLS or auth failures: verify certificate trust chain and MQTT credentials for the active environment.
